# UX-004 Use Motion To Explain State Continuity

Status: Draft
Domain: UX

## Source Research

- `research/ux/motion-as-state-continuity.md`

## Why

Users can lose orientation when objects change state, location, or level of detail. Purposeful motion can connect the before and after states, but generic animation adds delay and accessibility risk.

## Rule

Use motion only when it explains the continuity of an object, state, or navigation context. Define its start state, end state, reusable motion role, reduced-motion behavior, and unsupported-platform fallback.

## Good

- A selected item visibly becomes its detail surface while the source context remains understandable.
- Insert, remove, reorder, expand, and collapse transitions clarify what changed.
- Repeated transitions use named timing and easing roles.
- Reduced motion reaches the same end state with the same focus and information.

## Bad

- Elements animate because the screen should feel more modern.
- Motion delays a primary action or prevents unrelated interaction.
- Every component uses unrelated durations or easing.
- The transition is required to understand success, error, or navigation.

## Review

- State the orientation problem the motion solves.
- Verify start, end, focus, interruption, and reduced-motion behavior.
- Confirm loading and error feedback remain available without animation.
- Check support and progressive fallback for any platform-specific API.
