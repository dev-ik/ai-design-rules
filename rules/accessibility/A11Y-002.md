# A11Y-002 Textual Input Error Recovery

Status: Draft
Domain: Accessibility

## Source Research

- `research/accessibility/textual-error-recovery.md`

## Why

Users cannot reliably recover when a form only changes color, hides the failed field, or says only that submission failed. Textual, field-specific feedback makes the failure understandable and actionable across visual and assistive-technology contexts.

## Rule

When input validation detects an error, identify the affected input and describe the error in text. Include a correction hint when the valid value or format is known; color and icons may reinforce, but must not be the only signal.

## Good

- “Email: enter an address in the format name@example.com” appears with the invalid email field.
- A form-level summary names each invalid field and links or moves focus to a recoverable field when appropriate.
- An error remains understandable after its visual error color is removed.

## Bad

- A field border turns red with no textual explanation.
- A generic “Something went wrong” message replaces field-specific validation feedback.
- A message says a value is invalid but provides no correction when the required format is known.

## Review

- Trigger every client-side and server-returned validation state.
- Verify that each message identifies the input and describes the failed condition in text.
- Verify that color, position, or animation is not the only error cue.
- Check that a known correction is explained without exposing sensitive validation details.
